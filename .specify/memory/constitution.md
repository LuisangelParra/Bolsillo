# Bolsillo ‚Äî Project Constitution

> Drop-in for `.specify/memory/constitution.md`. These are the **non-negotiable principles** every spec, plan, task list, and implementation MUST adhere to. Derived from `docs/PRD.md` and `docs/TRD.md`. A plan, task, or change that violates any article is **invalid** and MUST be revised before implementation. "MUST" / "MUST NOT" / "FORBIDDEN" are binding; "SHOULD" is a strong default that requires written justification to deviate from.

## Article I ‚Äî Local-first, offline by default
The **on-device database is the single source of truth**. All core features (recording, categorization, accounts, budgets, recurring, reports, backup/export) MUST function with **no network**. No financial data leaves the device by default ‚Äî the default count of financial bytes sent off-device is **zero**. Any cloud capability (backup, sync) MUST be **optional, opt-in, and end-to-end encrypted**; a server, if present, stores only encrypted blobs and has **zero knowledge** of content. Removing connectivity MUST NOT disable or degrade any core feature.

## Article II ‚Äî Privacy by design
No mandatory account, sign-up, or bank connection to use the product. **No PII collection.** AI inference and learning run **100% on-device**. Permissions are minimal, justified at runtime, and the app MUST remain fully functional if any permission is denied ‚Äî camera is used only for receipts, location only (optionally) to improve suggestions and is processed on-device. Telemetry is **anonymous, opt-in, and MUST NEVER include financial content**; model-quality metrics are computed locally and, only with opt-in, shared in aggregate/anonymous form. Crash reports MUST exclude PII.

## Article III ‚Äî Financial integrity is sacred
Monetary values MUST be stored and computed as **integer minor units** (`*_minor`) or a **fixed-precision Decimal** type. Using `float`/`double` for money is **FORBIDDEN**, including intermediate calculations. Rounding is **banker's half-up**, defined once and tested. Transfers are modeled as a **linked double entry** (`transfer_group_id`: one negative at source, one positive at destination) and are excluded from expense/income totals; **balances MUST always reconcile**. All writes are **transactional and idempotent** ‚Äî no write may leave a half state ‚Äî with WAL journaling and crash recovery so the DB is never corrupted. For multi-currency, `fx_rate` and `amount_base_minor` are **computed once at creation and frozen**; they are NEVER recomputed when later rates change. Minor-unit handling honors each currency's `decimal_digits`. **Nothing is ever hard-deleted on user action**: deletion sets `deleted_at` (soft delete + trash with configurable purge).

## Article IV ‚Äî Speed of recording is the #1 success metric
Recording an expense MUST be achievable in **‚â§ 3 taps and ‚â§ 5 seconds** (amount ‚Üí AI-presuggested category/account ‚Üí save), with the numeric keypad focused on app open. The following budgets are binding and MUST hold on a reference low-end device: **cold start ‚â§ 1.5 s**, **save a transaction ‚â§ 100 ms**, **categorization inference ‚â§ 200 ms**, **receipt OCR ‚â§ 2 s**, **language switch applied app-wide ‚â§ 1 s**, **app + base model ‚â§ ~150 MB** (model becomes a download if it grows). Recording MUST NOT show a loading screen. No design or implementation may regress these targets without explicit, recorded sign-off.

## Article V ‚Äî The AI suggests; the user decides
Categorization runs a fixed cascade: **(1) user rules ‚Üí (2) merchant dictionary ‚Üí (3) on-device ML classifier**. User rules are deterministic with confidence 1.0; the dictionary is updatable **as data, not as a model**. The classifier returns a calibrated `confidence` in `[0,1]` plus top-3 alternatives. The confidence threshold defaults to **0.75** and is configurable: at or above it, the category auto-applies **with undo**; below it, the item is flagged **"to confirm"** with one-tap top-3 options. Confidence is shown but unobtrusive. **The AI MUST NEVER block saving.** Every confirmation/correction creates an on-device label (`AiCorrection`) that feeds **on-device personalization only**; retuning is incremental, local, and runs in the background (while charging/connected). A base-model update MUST NOT overwrite the user's personalization.

## Article VI ‚Äî Native per platform; the spec is the shared source of truth
Android (**Kotlin / Jetpack Compose**) and iOS (**Swift / SwiftUI**) are implemented **natively and independently**; there is **no shared application code**. The **platform-neutral feature spec is the single shared source of truth.** If platforms diverge in behavior, the fix belongs in the **spec first**, then in both implementations ‚Äî divergence is a spec defect, not an implementation choice. The **only** shared artifacts are data assets: the category taxonomy, the merchant dictionary, i18n strings, the AI model (exported to **Core ML** and **LiteRT**, or **ONNX** for both), and the **shared design system**. Both platforms MUST consume the same shared asset files.

**Visual parity:** both native apps MUST share the **same design system** and faithfully replicate the **canonical design** (`docs/design/` + `shared-assets/design/`: `design-system.md` + `tokens.json`). Both MUST map the **same design tokens** into native theme resources (no literal colors/sizes in UI code) and match component layout, states, and behavior. **Visual divergence between platforms is a design-system defect**: it is fixed in the shared design system (and the canonical design) **first**, then reflected in both implementations ‚Äî never resolved unilaterally in one app.

## Article VII ‚Äî Localization and currency are first-class
Default UI language is **Spanish (`es`)**, switchable to **English (`en`)** app-wide and applied immediately (‚â§ 1 s, without restart where possible). **No hard-coded user-facing strings** ‚Äî every string, including system category and default account names, resolves from localized resources via **string keys**; the architecture MUST allow adding languages by dropping in resource sets with **no code changes**. **USD and COP ship preloaded, enabled, and flagged essential (non-removable)**, guaranteeing a working default; the base currency is configurable to any enabled currency, and other **ISO 4217** currencies are addable (and, if user-added, removable). Number, currency, and date formats follow the active locale and each currency's `decimal_digits`. The unified multi-currency view rolls totals up to the base currency while **preserving each item's original amount**.

## Article VIII ‚Äî Clean architecture and modularity
Strict layered separation: **presentation ‚Üí domain ‚Üí data**. The domain knows nothing of SQLite, SQLCipher, Core ML, LiteRT, or OCR ‚Äî only **ports (interfaces)**: at minimum `Repository`, `ExpenseClassifier`, `OCR`, `FxProvider`, `LocaleProvider`. The AI is accessed **behind the `ExpenseClassifier` port** and is replaceable without touching the domain. **The UI MUST NEVER touch the database directly.** Dependencies point inward; data-layer implementations are injected into the domain via the ports.

## Article IX ‚Äî Test-driven and no data loss
Tests accompany **every** feature and MUST cover: money arithmetic and rounding, FX conversion, repositories and **schema migrations**, AI cascade and fallbacks (rules/dictionary-only mode included), and critical UI flows (**3-tap recording**, budgets, recurring, **language switch es‚Üîen**), plus localization rendering in both `es` and `en` (no truncation, no hard-coded strings) and currency formatting. Schema migrations are **versioned, idempotent, and tested with sample data**. **Release is BLOCKED** if any migration loses data or is irreversible, if balances fail to reconcile in the suite, if UI coverage is incomplete in both `es` and `en`, or if performance budgets (Article IV) fail on the reference low-end device.

## Governance

This constitution **supersedes** all ad-hoc decisions; where any spec, plan, task, or code conflicts with it, the constitution wins and the other artifact MUST be corrected.

**Compliance is enforced at every Spec Kit phase:**

- **`/speckit-plan` ‚Äî MUST verify compliance.** Each plan (`plan-android.md`, `plan-ios.md`) MUST contain a *Constitution Compliance* section that explicitly checks all nine articles and states how the design satisfies each. A plan that violates, omits, or weakens any article is **invalid** and MUST be revised before `/speckit-tasks` or `/speckit-implement` may run.
- **`/speckit-analyze` ‚Äî MUST flag violations and MUST run before implementation.** It cross-checks the spec, plans, and task lists against all nine articles and emits **blocking findings** for any violation (e.g., `float` money, hard delete, UI touching the DB, missing es/en coverage, missing tests, perf-budget regressions, AI blocking save, financial data leaving the device, non-essential treatment of USD/COP, hard-coded strings). It MUST NOT pass while any unresolved violation exists.
- **`/speckit-implement` ‚Äî MUST NOT proceed with unresolved violations and MUST verify compliance.** Implementation may begin only after `/speckit-analyze` is clean. The implementer MUST confirm, in code and tests, the binding rules: minor-units/Decimal money (no floats), banker's half-up rounding, soft delete + reconciling transfers, ports-only domain with `ExpenseClassifier`, frozen `fx_rate`/`amount_base_minor`, on-device-only AI with the 0.75 cascade that never blocks saving, `es`/`en` localization with no hard-coded strings, USD/COP essential, and the Article IV performance budgets.

**Platform divergence:** any behavioral difference between Android and iOS is fixed in the **spec first** (Article VI), then reflected in both plans and implementations.

**Amendments:** changing or weakening any principle requires an explicit **version bump** below, a recorded rationale, and synchronization of `docs/PRD.md`, `docs/TRD.md`, and dependent specs/plans. Tightening wording or adding clarity that does not alter a principle's intent is a **patch**; principle changes are **major**.
