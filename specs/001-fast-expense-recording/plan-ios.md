# Implementation Plan (iOS): Fast expense recording

**Feature**: `001-fast-expense-recording` | **Platform**: iOS | **Date**: 2026-06-13
**Spec**: [spec.md](./spec.md) · **Constitution**: `.specify/memory/constitution.md`
**Companion artifacts**: [research-ios.md](./research-ios.md) · [data-model.md](./data-model.md) (shared) · [data-model-ios.md](./data-model-ios.md) · [contracts/ports-ios.md](./contracts/ports-ios.md) · [contracts/use-cases-ios.md](./contracts/use-cases-ios.md) · [quickstart-ios.md](./quickstart-ios.md)
**Parity reference**: [plan-android.md](./plan-android.md) — same behavior, native iOS mechanics (Article VI).

## Summary

Deliver US-2.1 / US-2.2 / US-2.7 natively on iOS: open straight into a focused amount keypad, pre-fill category + account from the `ExpenseClassifier` port (stub returns last-used until spec 003), and save an expense in ≤ 3 taps / ≤ 5 s fully offline. Income and same-currency transfers (linked double entry, excluded from totals), one-tap undo, edit with balance recompute, and soft-delete + restore. Money is integer minor units (`Money`, `Int`); **account balance is always derived** (initial + signed-sum of non-deleted legs via GRDB `ValueObservation`) so reconciliation holds by construction. Stack: Swift 5.10 + SwiftUI, Clean Architecture + **MVVM** with `@Observable` on Swift Concurrency, SPM packages `BolsilloDomain` / `BolsilloData` / `FeatureRecord` + one new presentation-only package `BolsilloDesignSystem`, SQLite via **GRDB with SQLCipher**, localization via `.xcstrings` (es default + en). The UI is built faithfully to `docs/design/Bolsillo.dc.html` by mapping `shared-assets/design/tokens.json` into a SwiftUI **Theme environment + token extensions** (see **Design mapping**); no literal colors/sizes in views, visual parity with Android.

## Technical Context

**Language/Version**: Swift 5.10 (Xcode 15+/16), deployment target iOS 17.0
**Primary Dependencies**: SwiftUI, Observation (`@Observable`), Swift Concurrency (async/await, actors, `AsyncStream`), GRDB.swift 6.29+ built with **SQLCipher** (encryption at rest), Security (Keychain). No third-party UI/DI frameworks — composition is hand-rolled (Article VIII).
**Storage**: SQLite via GRDB, encrypted with SQLCipher; 256-bit passphrase in the iOS Keychain (`AfterFirstUnlockThisDeviceOnly`) — see [research-ios.md](./research-ios.md) R1.
**Design source**: `shared-assets/design/tokens.json` + `shared-assets/design/design-system.md`; canonical pixels in `docs/design/Bolsillo.dc.html`. UI maps tokens into a SwiftUI theme — no literal colors/sizes in views (see **Design mapping**).
**Testing**: Swift Testing (`import Testing`) for domain/data/design-system unit tests; XCUITest for the 3-tap flow and es/en rendering; `swift test` (host build via the package `.macOS` floor) for the fast domain loop; SQLCipher open verified on simulator.
**Target Platform**: iOS 17+ (iPhone). Xcode project generated by **Tuist** from `Project.swift`; local code in the SPM package `Packages/Bolsillo`.
**Project Type**: Native mobile app (one SPM library product per layer under `ios/Packages/Bolsillo`, app target via Tuist).
**Performance Goals**: save ≤ 100 ms; cold start ≤ 1.5 s to usable keypad; suggestion pre-fill ≤ 200 ms (stub is instant); no loading screen during recording.
**Constraints**: 100% offline; integer minor units only (no `Double`/`Float`/`Decimal`-from-double for money); soft delete only; transactional + reconciling writes; es default + en, no hard-coded strings.
**Scale/Scope**: Single-user on-device DB; thousands of transactions per account expected at MVP; one recording screen + supporting use cases.

**Unknowns resolved in [research-ios.md](./research-ios.md)**: SQLCipher key management on iOS Keychain + SPM packaging path; derived balance via `ValueObservation`; float-free amount parsing; signed-amount ledger; `@Observable`/MVVM on Swift Concurrency; modular `.xcstrings` localization; undo mechanics; classifier stub; testing stack. No `NEEDS CLARIFICATION` remain (spec clarifications closed 2026-06-13). One integration item flagged for early validation: the exact GRDB+SQLCipher SPM wiring (research-ios R1) — a packaging detail, not a behavior gap.

## Approach by layer

### Domain (`BolsilloDomain`)
- **New `Account`** model: `id`, `name` (user text, not a localized key), `type` (cash/debit/credit/bank/savings/wallet), `currencyCode`, `initialBalance: Money`, `icon`, `color`, `archived`, timestamps. (Only the subset needed for recording; full account management is E3.)
- **Signed-amount ledger** (shared invariant, see [data-model.md](./data-model.md)): `Transaction.amount`/`amountBase` are **signed** — expense < 0, income > 0, transfer source < 0, destination > 0. Display uses magnitude. The existing `Money` already supports unary minus.
- **Balance is derived, never stored**: `balance(account) = initialBalance + Σ(signed amount of non-deleted legs)`. No cached balance ⇒ edit/undo/delete/restore reconcile automatically.
- **New `AccountRepository` port**; **extend `TransactionRepository`** with `lastUsed()`, atomic `upsertTransfer(legSource:legDest:)`, and `softDeleteGroup`/`restoreGroup` for transfer pairs (full signatures in [contracts/ports-ios.md](./contracts/ports-ios.md)).
- **Use cases** (one responsibility each, constructor-injected ports, `callAsFunction`): `RecordTransaction`, `RecordTransfer` (builds the linked pair, throws on same-account / cross-currency), `EditTransaction`, `SoftDeleteTransaction`, `RestoreTransaction`, `UndoLastRecord` (soft-deletes the just-created id/group within the undo window — nothing lost), `ObserveAccountBalances`, `SuggestCategoryAndAccount` (calls classifier; falls back to last-used; never throws to the caller). `MoneyParser` converts keypad digits → `Money` with pure integer math honoring `Currency.decimalDigits`. (Signatures in [contracts/use-cases-ios.md](./contracts/use-cases-ios.md).)

### Data (`BolsilloData`)
- **GRDB schema v1** via `DatabaseMigrator`: `accounts`, `transactions`, `categories`. `transactions` indexed on `(accountId, deletedAt)` and `transferGroupId`. Money columns are `INTEGER`; enums as `TEXT`. GRDB record types are separate from domain types; **mappers** keep GRDB out of `BolsilloDomain`. Persistence specifics in [data-model-ios.md](./data-model-ios.md).
- **SQLCipher**: open the `DatabaseQueue` through `Configuration.prepareDatabase { try $0.usePassphrase(key) }`; `KeychainKeyProvider` generates a 256-bit key once and stores it device-only in the Keychain (research-ios R1). Validate the GRDB+SQLCipher SPM wiring first (the one flagged integration item).
- **Repositories**: `GRDBTransactionRepository`, `GRDBAccountRepository` implement the (extended) ports; multi-row writes run inside one `write {}` for atomicity (Article III). Derived balance exposed via `ValueObservation` over the SUM query, bridged to `AsyncStream<Money>`.
- **Stub classifier** `LastUsedExpenseClassifier`: `suggest()` returns the last-used `categoryId` with `confidence = 0.0`, no alternatives; `learn()` is a no-op (E4). Account fallback handled by `SuggestCategoryAndAccount`.
- **Seeds**: `CurrencySeed` (exists, USD+COP essential). Add `AccountSeed` (a single default Cash account) so first launch has an account to record into; `CategorySeed` is minimal (a default "uncategorized" with a localized `nameKey`).

### Presentation (`FeatureRecord`)
- **MVVM + `@Observable`**: `RecordModel` is a `@MainActor @Observable` final class holding `RecordState`; the view calls intent methods (`amountChanged`, `selectType(.expense/.income/.transfer)`, `selectCategory`, `selectAccount`, `selectDestination`, `save`, `undo`, `dismissUndo`). Use-case calls are `async`; writes run off the main actor and results flow back via `await` + balance `AsyncStream`. One-shot effects (saved/undone/validationError) are exposed as a transient `RecordState.event` the view consumes and clears (research-ios R5).
- **Screen**: `RecordScreen` rebuilt from the placeholder — `AmountKeypad` (custom integer entry, auto-focus on open, no system keyboard), `TypeSelector` segmented control, category + account chips (pre-filled), `SaveButton` always enabled when amount > 0, `UndoToast` shown 5000 ms or until next action. Transfer mode reveals source + destination pickers with same-account inline error (`record.transfer.sameAccountError`).
- **3-tap path**: launch → (amount typed) → Save. Category/account already pre-filled ⇒ taps = digits + Save. Undo is a 1-tap affordance, not on the critical path.
- **Strings**: `FeatureRecord/Resources/Localizable.xcstrings` (es source + en) is **generated from the shared i18n source** `shared-assets/i18n/{es,en}.json` (Article VI/VII), resolved via `Bundle.module`. The dotted keys (`record.amount`, `record.save`, `record.undo`, `record.income`, `record.expense`, `record.transfer`, `record.transfer.source`, `record.transfer.destination`, `record.transfer.sameAccountError`, `record.category`, `record.account`, `record.edit`, `record.delete`, `record.restore`, `record.deleted.toast`, `record.saved.toast`, `record.toConfirm` (row badge), `record.ai.waiting`, `record.ai.toConfirm` (AI badge below-threshold), `record.ai.confident`, `record.keypad.hint`) match the shared JSON **verbatim** — no renaming (parity win; a sync step keeps the catalog in lockstep with the JSON).

### App (`Bolsillo` target)
- **Composition root** in `BolsilloApp`: build `BolsilloDatabase` (GRDB+SQLCipher), GRDB repositories, the stub classifier, and the use cases; inject them into `RecordModel`. Wrap the root in `BolsilloThemeProvider` (injects the theme by `colorScheme`) and present `RecordScreen` full-surface as the launch destination (FR1). App-level strings stay in `App/Resources/Localizable.xcstrings`.

## Design mapping

Faithful to `docs/design/Bolsillo.dc.html` via `shared-assets/design/{tokens.json,design-system.md}`. **No new visual decisions** — every value below is a token reference. `§N` = section in `design-system.md`. All literals (hex/pt) live only in the `BolsilloDesignSystem` theme generated from `tokens.json`; views read them through the **theme environment** (`@Environment(\.bolsilloTheme)`), never inline. Visual parity with Android's Compose theme is guaranteed because both map the same `tokens.json`.

### Theme (`BolsilloDesignSystem`) — tokens → SwiftUI
- **`BolsilloTheme`** is an immutable value (`struct`) grouping sub-namespaces (`colors`, `typography`, `spacing`, `radius`, `elevation`, `gradients`, plus `categoryColor(_:)` and `confidence(_:threshold:)`). It is injected via a custom `EnvironmentValues.bolsilloTheme` key. **`BolsilloThemeProvider`** reads `@Environment(\.colorScheme)` and injects the `light`/`dark` token set so every descendant resolves the correct variant. Convenience `Color`/`Font`/`CGFloat` token extensions are provided for previews, but production views read from the environment. (User option chosen: **Theme environment** as the primary mechanism, token extensions as ergonomics.)
- **Color** (`tokens.json → color`, §1.1). SwiftUI has no Material `ColorScheme` slot system, so **`BolsilloColors`** carries every semantic role as a `Color`: `background, surface, surfaceAlt, surfaceInverse, onSurfaceInverse, primary, primaryContainer, onPrimary, onPrimaryContainer, primaryAccent, textPrimary, textSecondary, textMuted, textDisabled, outline, divider, track, fill, sheetHandle, success(+Container,+on), danger(+Container,+on), warning(+Container,+Border,+on), info(+Container), amountPositive, amountNegative, notificationDot, caret`. Two instances (`.light`, `.dark`) are built from the token table; the provider picks by scheme. Brand gradients (`tokens.json → gradient`: hero/primary/fab/avatar/navScrim) are `LinearGradient` factories on `BolsilloGradients` (angles/stops from the token strings).
- **Typography** (`tokens.json → typography`, §1.3). Plus Jakarta Sans is bundled in the package `Resources` (weights 400/500/600/700/800) and registered at first use; `BolsilloTypography` exposes a `Font` per role (`displayBalance, displayAmount, moneyXL, moneyL, keypadDigit, titleXL, titleL, titleM, button, bodyStrong, amountRow, body, bodyValue, label, labelSmall, caption, badge, navLabel, overline`) via `Font.custom(_:size:)` with the role weight, `lineHeight`, and `letterSpacing` (`.tracking`/`.kerning`). Money/tabular roles (`displayAmount, displayBalance, moneyXL/L, amountRow`) apply **tabular figures** with `.monospacedDigit()` (§1.3 mandatory rule). Fallback to the system font if the custom family is unavailable.
- **Spacing** (`tokens.json → spacing`, §1.4): `BolsilloSpacing` `CGFloat` scale (`xxs 2 … xl 18 … 6xl 30`); screen padding 18, sheet 20 (`layout.screenPaddingX`/`sheetPaddingX`).
- **Shapes/Radii** (`tokens.json → radius`, §1.5): `BolsilloRadius` `CGFloat` set (`iconTileSm 11, iconTile 13, iconTileLg 17, control/chip 16, card 18, cardLarge 20, cardXL 22, nav 26, sheet 30, frame 46`); `full` → `Capsule()`. Applied via `RoundedRectangle(cornerRadius:)` / `.clipShape`. (User option: Shape extensions provided alongside the theme.)
- **Elevation/shadow** (`tokens.json → elevation`, §1.6): `BolsilloElevation` maps each token (`e1…e4, key, nav, fab, buttonPrimary, segmentedThumb, toast`) to a `.shadow(color:radius:x:y:)` preset; ambient color is dark-violet `rgba(28,20,60,a)` in light, `rgba(0,0,0,a)` in dark. FAB/Save carry the violet-tinted glow; cards stay low and soft.
- **Category** (`tokens.json → category`, §1.2): `categoryColor(_ token:) -> (fg: Color, container: Color)` keyed by the **palette token id** (13 ids, light/dark container). A transaction's `categoryId` is a **taxonomy** id, so resolve color as `taxonomy[categoryId].colorToken → categoryColor(token)` (`shared-assets/taxonomy/category-taxonomy.json` carries `colorToken`); never key the palette directly off a taxonomy id. **Confidence** (`tokens.json → confidence`, §2.3): `confidence(_ conf:, threshold:)` → a `ConfidenceVisual { icon, fg, container, labelKey, chipBorder? }` for `waiting | toConfirm | confident`, keyed off `ExpenseClassifier.defaultThreshold` (0.75), not a literal.
- **`MoneyText`** (`BolsilloDesignSystem/Components`): renders an `Int` minor-unit amount + `decimalDigits` + currency symbol + locale + sign, **tabular** (`.monospacedDigit()`), colored `amountPositive`/`amountNegative` per §1.1. Takes **primitive inputs** (no domain `Money` import) so the package stays dependency-free; `FeatureRecord` adapts `Money` → primitives at the call site. The single money renderer used everywhere.

## Testing strategy (Article IX)
- **`BolsilloDomain` (Swift Testing, host)**: `Money` arithmetic; `MoneyParser` integer parsing per `decimalDigits`; `RecordTransaction` balance effect; `RecordTransfer` builds the signed linked pair, excluded from expense/income totals, throws on same account, throws on cross-currency; `EditTransaction` recompute incl. account change + transfer leg; `SoftDelete`/`Restore` balance effect; `UndoLastRecord`; **reconciliation property test** over random op sequences; overdraft allowed. Runs via `swift test`.
- **`BolsilloData` (Swift Testing)**: GRDB record insert/query with `deletedAt` filtering; balance SUM query correctness; transactional transfer-write atomicity; **migration test with sample data** (v1 baseline via `DatabaseMigrator`; release blocked on any data loss).
