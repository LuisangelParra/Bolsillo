# Phase 0 — Research (iOS): Fast expense recording

Decisions that resolve the iOS plan's unknowns. Format: Decision · Rationale · Alternatives rejected.
Behavior is shared with Android (the spec is platform-neutral, Article VI); only the iOS *mechanics* differ. Where a decision matches the Android one, this notes it so the two implementations stay aligned.

## R1 — SQLCipher key management under GRDB
- **Decision**: Open the GRDB `DatabaseQueue` with SQLCipher enabled via a `Configuration.prepareDatabase { db in try db.usePassphrase(key) }` hook. The passphrase is a random 256-bit key generated once on first launch with `SecRandomCopyBytes`, stored in the **iOS Keychain** under `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` (device-only, not synced to iCloud). The unwrapped key lives in memory only while the queue is open.
- **Rationale**: Mirrors the Android decision (random key wrapped by hardware-backed keystore) — the DB file is useless if copied off-device (Articles I/II). `AfterFirstUnlockThisDeviceOnly` keeps background DB access working (e.g. cold launch) without exporting the key. No user secret needed for MVP recording.
- **SPM packaging caveat (the one early-validation item)**: GRDB's plain SwiftPM product builds against the system SQLite, which has **no codec** — SQLCipher is not active by default. Enabling it requires building GRDB against a SQLCipher-provided `sqlite3` with `SQLITE_HAS_CODEC`. Resolve the exact mechanism in `tasks-ios.md`, in this preference order: (a) a SQLCipher SwiftPM module the data target links so GRDB compiles with the codec; (b) a prebuilt SQLCipher `.xcframework` binary target; (c) CocoaPods `GRDB.swift/SQLCipher` subspec only if SPM proves impractical (last resort — the project is SPM/Tuist-first). **Validate this integration first**, before building features on top, because it changes the `Package.swift` GRDB wiring. The key-management policy above is decided regardless of which mechanism wins.
- **Alternatives rejected**: device-id-derived passphrase (extractable); requiring a user PIN/biometric to open the DB (blocks fast recording — app lock is a later feature); storing the key in `UserDefaults`/plist (not encrypted at rest); iCloud-synced Keychain item (key would leave the device — violates Article I/II).

## R2 — Derived balance via GRDB ValueObservation
- **Decision**: Balance is **always derived** — `initialBalance + SUM(signed amount of non-deleted legs)` from an indexed SQL aggregate, never a stored `balance` column. Expose it reactively with GRDB **`ValueObservation`** bridged to Swift Concurrency (`observation.values(in:)` → `AsyncSequence`, consumed on the `@MainActor` `@Observable` model). Matches Android's derived-balance decision.
- **Rationale**: Reconciliation (FR 15) holds **by construction** — edit/undo/delete/restore mutate rows and the SUM follows; it cannot drift. `ValueObservation` recomputes only on writes that touch the observed rows and delivers on the main actor, fitting `@Observable`. At MVP scale a SUM over one account's indexed rows is well under the 100 ms budget.
- **Alternatives rejected**: cached running-balance column (must be recomputed transactionally on every mutation; any missed path silently breaks reconciliation — the exact bug Article III forbids); polling/manual refresh (misses external writes, fights the reactive model). Revisit caching only if profiling on the reference device shows the SUM exceeds budget at realistic row counts.

## R3 — Float-free amount entry
- **Decision**: Custom integer keypad + a `MoneyParser` that accumulates digits into `Int` minor units using `Currency.decimalDigits` (digit placement is pure integer math). Display formatting runs `Foundation.FormatStyle` / `NumberFormatter` (currency style, active locale) over the **integer** value; parsing never passes through `Double`/`Decimal`-from-double.
- **Rationale**: Article III forbids float for money including intermediate steps; `Money` exposes no floating accessor (keep it that way). A digit-accumulator keypad also removes locale decimal-separator ambiguity and is faster (fewer taps) — supports Article IV. Matches Android.
- **Alternatives rejected**: SwiftUI `TextField` bound to a `Double`/`Decimal(string:)` (float path, locale-fragile); `Decimal` everywhere (heavier than needed — `Int` minor units is the constitution's primary representation and is already the `Money` type).

## R4 — Signed-amount ledger convention
- **Decision**: Store `Transaction.amount` and `amountBase` as **signed** minor units: expense < 0, income > 0, transfer source leg < 0, destination leg > 0. The UI displays magnitude (sign drives color via `amountPositive`/`amountNegative`). Identical to Android — keeps both platforms' data semantics aligned (Article VI).
- **Rationale**: Makes balance a trivial `SUM` (R2) and makes the transfer double entry self-balancing (`legSource.amount + legDest.amount == 0`). Aligns with spec FR 9 ("one negative at source, one positive at destination"). The current `Money` struct already supports unary minus.
- **Alternatives rejected**: magnitude-only storage with sign re-derived from `type`/leg at query time (every aggregate needs CASE logic; SUM can't be a plain column sum — error-prone).

## R5 — Presentation: @Observable + MVVM on Swift Concurrency
- **Decision**: One `@MainActor @Observable` model per screen (`RecordModel`) holding `RecordState` and exposing intent methods (`amountChanged`, `selectType`, `save`, `undo`, …). It calls domain use cases with `async`/`await`; repository writes run off the main actor (GRDB queue / actor-isolated data layer) and results flow back via `ValueObservation` and awaited returns. One-shot effects (saved/undone/validation-error) are surfaced as transient state the view observes (e.g. an enum `transientEvent` cleared after presentation), not a Combine subject.
- **Rationale**: `@Observable` (Observation framework, iOS 17) is the current idiomatic replacement for `ObservableObject`/`@Published` — fine-grained, less boilerplate, and the user asked for it. `@MainActor` keeps UI state mutations on the main thread; `await` keeps Save off the UI-blocking path (Article IV). MVVM (not Android's MVI) is the platform-idiomatic choice the user specified; behavior parity is preserved because both consume the same use cases.
- **Alternatives rejected**: `ObservableObject` + `@Published` (legacy boilerplate, coarser invalidation); putting logic in the `View` (untestable, violates separation); a Redux/TCA store (heavier than needed for one screen, and diverges from Android's structure without behavioral benefit).

## R6 — Modular localization with String Catalogs (.xcstrings)
- **Decision**: Each presentation module owns a `Localizable.xcstrings` in its target `resources`, resolved via the module bundle: `Text("record.amount", bundle: .module)` / `String(localized: "record.amount", bundle: .module)`. `FeatureRecord` gains its own catalog (es source + en) keyed by the **dotted** spec §9 keys (`record.amount`, `record.save`, `record.undo`, …) — which match the existing app catalog keys (`record.save`, `record.title`) exactly. The app target keeps its app-level catalog. `defaultLocalization: "es"` (already set) makes es the development/source language.
- **Rationale**: No hard-coded user-facing strings (Article VII); adding a language is dropping in a translation with no code change. Dotted keys are the spec's canonical key form and need no Android-style renaming, so the two platforms key off the *same* string ids — a parity win (Article VI). The current `RecordView` placeholder injects strings via `init` params; that doesn't scale to a full screen, so the feature moves to a module-owned catalog.
- **Alternatives rejected**: keeping all strings in the app target and passing a strings struct into `FeatureRecord` (couples the feature to the app, awkward for previews/tests); hard-coded Spanish with runtime lookup (violates Article VII); per-string `init` injection (the scaffold placeholder approach — unmaintainable at feature scale).

## R7 — Undo mechanics
- **Decision**: After a save, keep the new transaction id (and, for transfers, the `transferGroupId`) in `RecordState`. **Undo = soft-delete** that record/group; the derived balance reverts automatically (R2). The affordance is a custom **undo toast** shown 5000 ms (token `motion.undoToastTimeout`) or until the next user action (FR 6). No hard delete (Article III) — undone items are recoverable like any trashed item.
- **Rationale**: One code path for undo and delete (soft delete), so reconciliation and "no data loss" hold uniformly; matches the clarified undo window and the Android decision.
- **Alternatives rejected**: buffer the write and only commit after the toast times out (violates "save ≤ 100 ms" perception and "balance updates immediately"); hard-delete on undo (violates Article III).

## R8 — Classifier stub behavior (until spec 003)
- **Decision**: `LastUsedExpenseClassifier.suggest()` returns the last-used `categoryId` with `confidence = 0.0` and empty alternatives; `learn()` is a no-op `async` method. Account fallback (last-used, else the seeded onboarding account) is handled in `SuggestCategoryAndAccount`. Matches Android.
- **Rationale**: 001 only *consumes* a suggestion (spec §3 out-of-scope: AI internals + correction logging are E4). Confidence/"to-confirm" UI is built (design parity) but driven by the token-defined states; Save is never gated on the classifier (Article V). The `ExpenseClassifier` port already exists in `BolsilloDomain`.
- **Alternatives rejected**: returning `confidence = 1.0` (implies a real user-rule-grade suggestion); implementing the full cascade now (out of scope, belongs to E4 / spec 003).

## R9 — Testing stack
- **Decision**: Use **Swift Testing** (`import Testing`, `@Test`/`#expect`) for new domain, data, and design-system unit tests (the scaffold already builds the Testing module; `MoneyTests`/`SmokeTests` exist). Use **XCUITest** for the 3-tap flow and es/en rendering UI tests in the app target. Fast domain loop via `swift test` from `Packages/Bolsillo/` (host build through the `.macOS` floor); full build/UI via `xcodebuild ... -destination 'platform=iOS Simulator'`.
- **Rationale**: Swift Testing is the modern idiom and is already wired; XCUITest is required for a real 3-tap launch-to-save assertion and on-device SQLCipher. Article IX requires money math, transfer integrity, reconciliation, migration-with-sample-data, and a 3-tap UI test in es+en.
- **Alternatives rejected**: XCTest-only (works, but Swift Testing is cleaner and already present); ViewInspector/snapshot-only UI checks (don't prove the real launch path or DB encryption).
