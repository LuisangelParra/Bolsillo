# Implementation Plan (iOS): Onboarding and initial setup

**Feature**: `002-onboarding` | **Platform**: iOS | **Date**: 2026-06-13
**Spec**: [spec.md](./spec.md) · **Constitution**: `.specify/memory/constitution.md`
**Companion artifacts**: [research-ios.md](./research-ios.md) · [data-model.md](./data-model.md) · [contracts/ports-ios.md](./contracts/ports-ios.md) · [contracts/use-cases-ios.md](./contracts/use-cases-ios.md) · [quickstart-ios.md](./quickstart-ios.md)
**Parity reference**: [plan-android.md](./plan-android.md) — same behavior, native iOS mechanics (Article VI).

## Summary

Deliver US-1.1 / US-1.2 / US-1.3 natively on iOS: a new user, with no account / no email / no network, reaches the record surface (feature 001) in ≤ 60 s. Onboarding picks the **base currency** (USD/COP pinned at top + "choose another" ISO 4217), creates **exactly one** first account (type `cash`, name key `account.defaultName.cash`, currency = base, initial balance default `0`), silently persists `language = es`, and writes an **atomic** onboarding-complete marker so subsequent launches go straight to recording. Reuses 001's `BolsilloDomain` (`Account`, `Currency`, `AccountRepository`) and `BolsilloData` (GRDB schema v1, SQLCipher, `CurrencySeed` with USD+COP essential). New `FeatureOnboarding` SwiftUI package orchestrated by a `@MainActor @Observable` `OnboardingModel`. A new `SettingsRepository` port lives in `BolsilloDomain` and is backed by a GRDB-backed `app_settings` table — so the four onboarding writes (`base_currency`, `language`, first account, `onboarding_completed_at`) land in **one** GRDB `write {}` block, satisfying Article III atomicity. UI maps directly to `docs/design/Bolsillo.dc.html` via the existing `BolsilloDesignSystem` package (`shared-assets/design/tokens.json` + `design-system.md`); no literal colors/sizes in views, visual parity with Android.

## Technical Context

**Language/Version**: Swift 5.10 (Xcode 15+/16), deployment target iOS 17.0
**Primary Dependencies**: SwiftUI, Observation (`@Observable`), Swift Concurrency (`async/await`, actors, `AsyncStream`), GRDB.swift 6.29+ built with **SQLCipher** (encryption at rest, from 001), Security (Keychain, from 001). No third-party UI/DI frameworks — composition is hand-rolled (Article VIII).
**Storage**: Reuses 001's encrypted GRDB database. Adds one new table `app_settings(key TEXT PRIMARY KEY, value TEXT NOT NULL)` via a v2 migration registered in `DatabaseMigratorV2.swift`.
**Design source**: `shared-assets/design/tokens.json` + `shared-assets/design/design-system.md`; canonical pixels in `docs/design/Bolsillo.dc.html`. All onboarding views consume `BolsilloThemeProvider` from `BolsilloDesignSystem` — no literal colors/sizes (see **Design mapping** below).
**Testing**: Swift Testing (`import Testing`) for domain/data unit tests; XCUITest for defaults path, currency order, USD/COP non-removable, no-language-selector, re-launch routing; `swift test` (host build via the package `.macOS` floor) for the fast domain loop; GRDB in-memory + sample-data migration test on simulator.
**Target Platform**: iOS 17+ (iPhone). Xcode project generated by **Tuist** from `Project.swift`; local code in the SPM package `Packages/Bolsillo`.
**Project Type**: Native mobile app (one SPM library product per layer under `ios/Packages/Bolsillo`, app target via Tuist).
**Performance Goals**: cold start ≤ 1.5 s into the onboarding welcome screen; onboarding-complete atomic write ≤ 100 ms (Article IV); defaults-path wall-clock ≤ 60 s (spec §10); no loading screen mid-flow.
**Constraints**: 100% offline; integer minor units only (initial balance is `Money(minorUnits: Int)`); soft delete only on any later changes (E3); transactional + reconciling writes; `es` silently persisted, en switchable later (E12); no hard-coded strings.
**Scale/Scope**: Single-user device; one new package product + one new domain port + one schema migration.

**Unknowns resolved in [research-ios.md](./research-ios.md)**: settings-store choice (GRDB `app_settings` vs `UserDefaults`), atomic write strategy, first-launch routing signal, currency-list-order data shape, locale-derived default base currency, SwiftUI locale-application strategy. No `NEEDS CLARIFICATION` remain.

**Structure Decision**: Add **one** new SPM library product, `FeatureOnboarding`, mirroring `FeatureRecord`'s shape (one `@Observable` model, one entry view, screens under `Views/`). Settings persistence lives in **GRDB** (new `app_settings` table) rather than `UserDefaults` so the four onboarding writes share a single transaction — see [research-ios.md](./research-ios.md) decision D1. New domain `SettingsRepository` port keeps `FeatureOnboarding` from depending on `BolsilloData`. Existing `AccountSeed.seed(_:)` is **kept** (callable for tests / future fallback) but **`CompositionRoot.init()` no longer invokes it on startup** — onboarding owns first-account creation now (see [research-ios.md](./research-ios.md) D8). `CurrencySeed` is unchanged: USD + COP remain essential and pre-loaded. Must add `FeatureOnboarding` to `Package.swift` (`.library` product + target + test target) and to `Project.swift` app dependencies.

## Approach by layer

### Domain (`BolsilloDomain`)
- **No new entity types**: reuses `Account`, `Currency`, `Money`, `AccountType` from 001.
- **New port `SettingsRepository`** (full signature in [contracts/ports-ios.md](./contracts/ports-ios.md)):
  - `func getBaseCurrency() async -> String?`
  - `func observeBaseCurrency() -> AsyncStream<String?>`
  - `func getLanguage() async -> String?`
  - `func isOnboardingComplete() async -> Bool`
  - `func observeOnboardingComplete() -> AsyncStream<Bool>`
  - `func completeOnboarding(baseCurrency: String, language: String, firstAccount: Account, enableCurrencyIfMissing: Currency?) async throws` — **atomic**, all-or-nothing. Implementation wraps every write in a single GRDB `write {}` block.
- **New port `LocaleProvider`**: `func regionCode() -> String?` — pure abstraction over `Locale.current.region?.identifier`. Implementation lives in `BolsilloData/LocaleProviderImpl.swift` so the domain stays Foundation-free at the port boundary.
- **New use cases** (one responsibility each, plain `struct`s with port deps via init):
  - `IsOnboardingComplete(settings:)` → `AsyncStream<Bool>` for app routing.
  - `GetEnabledCurrencies(currencies:)` → returns the list with **USD pinned at index 0, COP at index 1**, then remaining enabled currencies sorted by code. Pinning logic centralized here so the UI never has to know.
  - `EnableCurrency(currencies:)` → if the picked ISO 4217 code is not already in the catalog, insert it `isEnabled = true, isEssential = false`; if present, mark `isEnabled = true`. Returns the resolved `Currency`. Idempotent. Never flips `isEssential`.
  - `ResolveDefaultBaseCurrency(locale:)` → returns `"COP"` if `locale.regionCode() == "CO"`, else `"USD"` (spec §12 assumption locked in research-ios D5).
  - `CompleteOnboarding(settings:currencies:clock:)` → orchestrates: validates inputs (currency code non-empty / known; account currency == base; initial balance ≥ 0 minor units; language == "es"), then calls `settings.completeOnboarding(...)`. Never tries to recover from a partial state — atomicity is the repository's contract.

### Data (`BolsilloData`)
- **Schema v2 migration** (`DatabaseMigratorV2.register(in:)`):
  ```swift
  migrator.registerMigration("v2_app_settings") { db in
      try db.create(table: "app_settings") { t in
          t.primaryKey("key", .text)
          t.column("value", .text).notNull()
      }
  }
  ```
  Additive only — no data loss. Registered after V1 in `BolsilloDatabase.makeMigrator()`. Tested with sample rows in existing tables (Article IX).
- **`AppSettingRecord`** (`Codable`, `FetchableRecord`, `PersistableRecord`): `key: String`, `value: String`.
- **`GRDBSettingsRepository`** implements `SettingsRepository`. `completeOnboarding(...)` runs inside a single `db.queue.write { dbq in ... }`:
  1. Upsert `AppSettingRecord(key: AppSettingKeys.baseCurrency, value: baseCurrency)`
  2. Upsert `AppSettingRecord(key: AppSettingKeys.language, value: language)`
  3. If `enableCurrencyIfMissing != nil`, upsert its `CurrencyRecord`
  4. Insert `AccountRecord(firstAccount.toRecord())`
  5. Upsert `AppSettingRecord(key: AppSettingKeys.onboardingCompletedAt, value: ISO 8601 now)`

  Any thrown error inside the block aborts the write ⇒ all five reverted ⇒ next launch sees no marker ⇒ onboarding re-shown clean.
- **`Iso4217Catalog`** (NEW static bundle, Swift): list of `(code, displayNameKey, decimalDigits)` for ISO 4217 currencies the user can pick via "choose another". USD/COP excluded from this list (they live in `CurrencySeed`). Display names resolve through localized keys (`currency.name.<code>`) — never hard-coded.
- **`CurrencyRepository` extension**: add `func getIso4217Candidates() -> [Currency]` (sync, reads bundled catalog).
- **`CompositionRoot` change**: keep `CurrencySeed.essentials` insert via `BolsilloDatabase` first-open (so USD/COP always present). **Remove** the unconditional `try AccountSeed.seed(db)` call on `CompositionRoot.init()` — onboarding now creates the first account. Keep `try CategorySeed.seed(db)` (categories needed for recording, independent of onboarding).

### Presentation (`FeatureOnboarding`)
- **`@MainActor @Observable` model**: `OnboardingModel` exposes `state: OnboardingState` (immutable `struct` rebuilt on each mutation); intents are plain methods (`onContinueFromWelcome()`, `onUseDefaults()`, `onCurrencySelected(_:String)`, `onOpenIsoPicker()`, `onIsoCurrencySelected(_:String)`, `onCloseIsoPicker()`, `onInitialBalanceChanged(_:String)`, `onConfirmFirstAccount()`, `onBack()`); emits a one-shot `OnboardingEvent.finishedToRecord` via an `AsyncStream<OnboardingEvent>` (or a `@Published`-equivalent via a `let events: AsyncStream<OnboardingEvent>` exposed for the app shell to observe). State machine:
  - `Step.welcome` → `Step.currency` → `Step.firstAccount` → (atomic write) → `.finishedToRecord`.
  - `onUseDefaults()` from welcome skips currency + first-account: resolves default base via `ResolveDefaultBaseCurrency`, builds a Cash `Account` with `initialBalance = Money(minorUnits: 0, decimalDigits: baseCurrency.decimalDigits)`, calls `CompleteOnboarding` directly.
- **Views**:
  - `WelcomeView` — hero brand, "Continuar" + "Usar valores predeterminados" CTAs (resolved from `onboarding.cta.continue` / `onboarding.cta.useDefaults`). No language picker (spec FR7).
  - `CurrencyPickerView` — vertical `ScrollView` + `LazyVStack` (or `List` styled to match design). **USD row at index 0, COP row at index 1**, both with essential badge (`onboarding.currency.essential.badge`). "Elegir otra moneda" row presents `IsoCurrencyPickerSheet` via `.sheet`. Selecting an ISO row enables that currency and returns to the picker with it selected.
  - `FirstAccountView` — fixed-type "Efectivo / Cash" header (resolved from `account.defaultName.cash`); no name field on the defaults path (spec FR5 clarification). Amount entry uses the **same `AmountKeypad` + `AmountDisplay` views from `FeatureRecord`** — extracted to `BolsilloDesignSystem` if not already there (planned in tasks). Currency caption shows the selected base. Confirm CTA writes.
  - `IsoCurrencyPickerSheet` — search + scrollable list of `Iso4217Catalog` entries; tap to enable and dismiss.
  - `OnboardingStepDots` — design progress indicator.
- **Strings**: `Sources/FeatureOnboarding/Resources/Localizable.xcstrings` generated from `shared-assets/i18n/{es,en}.json` (same generation step as 001's `FeatureRecord/Resources/Localizable.xcstrings`, just with new dotted keys). New keys: `onboarding.welcome.title`, `onboarding.welcome.subtitle`, `onboarding.cta.continue`, `onboarding.cta.useDefaults`, `onboarding.currency.title`, `onboarding.currency.essential.badge`, `onboarding.currency.chooseAnother`, `onboarding.account.title`, `onboarding.account.initialBalance.label`, `onboarding.account.confirm`, `account.defaultName.cash` (es: "Efectivo", en: "Cash"), `currency.name.USD`, `currency.name.COP`, plus `currency.name.<code>` for each supported ISO. Resolved via `Bundle.module`.

### App shell (`ios/App/Sources/BolsilloApp.swift`, `CompositionRoot.swift`)
- `CompositionRoot` (extended): build `GRDBSettingsRepository`, `LocaleProviderImpl`, `IsOnboardingComplete`, `GetEnabledCurrencies`, `EnableCurrency`, `ResolveDefaultBaseCurrency`, `CompleteOnboarding`. Stop calling `try AccountSeed.seed(db)` on init.
- `BolsilloApp.body` decides root view via a `@State private var onboardingComplete: Bool? = nil` populated from `root.isOnboardingComplete()` on appear; while `nil`, show a single neutral background (no spinner per Article IV). Once resolved:
  - `false` ⇒ `OnboardingView(model: OnboardingModel(...))` (with `.environment(\.locale, Locale(identifier: "es"))` so the welcome screen renders in `es` even on an English device).
  - `true` ⇒ `RecordView(model: RecordModel(...))` (unchanged from 001).
- After `OnboardingEvent.finishedToRecord` (consumed from the model's events stream), the app shell swaps root to `RecordView` and sets `.environment(\.locale, Locale(identifier: persistedLanguage ?? "es"))` for the whole tree. On subsequent cold starts the saved marker routes straight to `RecordView` ⇒ onboarding is never shown again.
- **Locale strategy**: iOS does not have an app-wide `AppCompatDelegate.setApplicationLocales` equivalent without `CFBundleDevelopmentRegion` indirection. We use SwiftUI's `.environment(\.locale, ...)` set at the app root from the persisted `language` value (`es` always in 002). E12 will swap this dynamically. The persisted `language` setting is the single source of truth; no `UserDefaults` parallel.

## Testing strategy (Article IX)

- **`BolsilloDomainTests` (Swift Testing, `import Testing`)**:
  - `GetEnabledCurrencies` always returns USD at 0, COP at 1 regardless of input order, with the rest sorted alphabetically after.
  - `EnableCurrency` is idempotent and never re-essential-flags a non-essential currency; throws on unknown codes.
  - `ResolveDefaultBaseCurrency` returns COP for `regionCode == "CO"` and USD otherwise (incl. nil).
  - `CompleteOnboarding` rejects invalid inputs (currency code empty, account currency != base, balance negative).
- **`BolsilloDataTests` (Swift Testing on simulator)**:
  - `AppSettingRecord` round-trip (insert/fetch/update via `GRDBSettingsRepository`).
  - **Atomicity test**: `GRDBSettingsRepository.completeOnboarding(...)` — inject a failure (e.g., a poisoned `AccountRecord` rejected by a check constraint) at step 4 and assert that `base_currency`, `language`, the currency upsert, and the completion marker were **all** rolled back; the next `isOnboardingComplete()` returns `false`.
  - **Migration v1→v2 test**: seed a v1 DB with sample accounts/transactions/categories, run `DatabaseMigratorV2`, assert `app_settings` table exists, all prior rows intact, balances still reconcile (Article IX).
- **`FeatureOnboardingTests` (Swift Testing, `@MainActor`)**: `OnboardingModel` state transitions for defaults path; currency selection; back/forward; event emission on finish.
- **`BolsilloUITests` (XCUITest)** — every spec §6 scenario:
  - Defaults path: launch → tap "Usar valores predeterminados" → land on record surface; assert (a) marker is set, (b) base currency = USD or COP per device region, (c) exactly one account exists with type `cash`, balance `0`, name resolves to "Efectivo" (es) / "Cash" (en), currency = base.
  - Currency order: assert USD row at index 0, COP at index 1, both with essential badge, neither removable. Add EUR via "choose another" → EUR appears at index 2 (or after USD/COP).
  - Pick COP path: select COP → confirm → assert `base_currency == "COP"` and first account currency == COP.
  - Pick another ISO path: open ISO sheet → select EUR → confirm → assert EUR is now `isEnabled = true, isEssential = false`, base = EUR, first account currency = EUR, and USD + COP still present + essential.
  - No language selector: enumerate every accessibility identifier on every onboarding screen, assert none matches `onboarding.languagePicker.*`.
  - Re-launch: after a successful run, terminate + relaunch the app → assert the record surface is the start view and onboarding is never shown.
  - Crash-mid-flow recovery: inject a failure during `completeOnboarding(...)` (via a debug-only override in the composition root) → assert no partial DB state and onboarding restarts cleanly.
